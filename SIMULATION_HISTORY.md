## Step 1 — Build LLMServingSim's external simulator (astra-sim)

LLMServingSim is the **outer** scheduler/serving simulator. It needs the
**inner** discrete-event simulator (astra-sim) to advance simulation time per
iteration. astra-sim's analytical binary is the executable LLMServingSim
forks via subprocess pipes.

**Sub-step 1a: Clone astra-sim into the simulator tree**

```bash
cd LLMServingSim_ispass26
git clone --recursive https://github.com/casys-kaist/astra-sim.git
```

This pulled astra-sim and its own submodules: `chakra`, `fmt`, `spdlog`,
`yaml-cpp`, googletest, csg-htsim, ns-3, the analytical memory/network
backends.

**Sub-step 1b: Build inside author's docker image**

System cmake unavailable, so used author's docker image `astrasim/tutorial-micro2024`.
Started a long-lived container so dep installs persist:

```bash
sudo docker run -d --name servingsim_run \
  -v /home/jhpark/pjh_project/Lens-baseline/LLMServingSim_ispass26:/app/LLMServingSim \
  -w /app/LLMServingSim astrasim/tutorial-micro2024 sleep infinity

sudo docker exec servingsim_run bash -c "
  pip3 install pyyaml pyinstrument transformers datasets msgspec scikit-learn \
    xgboost==3.1.2 matplotlib==3.5.3 pandas==1.5.3 numpy==1.23.5
  pip3 install astra-sim/extern/graph_frontend/chakra
  bash astra-sim/build/astra_analytical/build.sh
"
```

Result: built binary
`astra-sim/build/astra_analytical/build/AstraSim/bin/AstraSim_Analytical_Congestion_{Aware,Unaware}`.
This is the binary `main.py` of LLMServingSim spawns as subprocess.

## Step 2 — Run 9-cell baseline sweep

Inside the long-lived container, ran the baseline 3 datasets × 3 batch sizes:

```bash
sudo docker exec -d servingsim_run bash -c "
  cd /app/LLMServingSim
  for ds in sharegpt cnn writing_prompts; do
    for bs in 1 2 4; do
      python3 main.py \
        --cluster-config cluster_config/single_node_tpuv6e_8b.json \
        --dataset dataset/v6e_8b_baseline/${ds}.jsonl \
        --output output/v6e_8b_baseline/${ds}_bs${bs}.csv \
        --max-batch ${bs} \
        --max-num-batched-tokens 2048 \
        --num-req 50 \
        --log-level WARNING
    done
  done
"
```

What ran inside each `python3 main.py` invocation:

1. **LLMServingSim core** (`main.py`, `inference_serving/*.py`)
   - Loads cluster_config, dataset JSONL, perf_models CSVs.
   - Spawns one **astra-sim subprocess** (the binary built in step 1).
2. **Per-iteration loop**:
   - LLMServingSim's `Scheduler` decides batch composition (which requests
     to prefill / decode this step).
   - For each batch, `trace_generator.py` looks up per-layer latency from
     `llm_profile/perf_models/TPU-v6e-1/meta-llama/Llama-3.1-8B/tp1/{layers,attention}.csv`,
     synthesizes a chakra ET workload trace, writes it to
     `astra-sim/inputs/workload/TPU-v6e-1/meta-llama/Llama-3.1-8B/instance0_batchK/`.
   - Spawns `python -m chakra.src.converter.converter LLM ...` per iteration
     to convert the textual workload to chakra ET protobuf.
   - Sends the workload path over the astra-sim subprocess's stdin.
   - astra-sim runs the iteration, prints `sys[0] iteration N finished, X cycles`
     to stdout.
   - Controller reads that line; LLMServingSim's clock advances by X cycles.
3. **End-of-run**: writes per-request CSV to `--output` path with columns
   `instance id,request id,model,input,output,arrival,end_time,latency,
   queuing_delay,TTFT,TPOT,ITL`.

Wall-time per cell:
- sharegpt bs=1: ~17 min (15,200 sched iterations)
- sharegpt bs=2: ~13 min
- sharegpt bs=4: ~6 min
- cnn bs=1: ~5 min, bs=2: ~3 min, bs=4: ~1 min
- writing_prompts bs=1: ~50 min (35,200 iterations, longest decode)
- writing_prompts bs=2: ~25 min, bs=4: ~10 min

Total ~2 h.

Outputs written to `LLMServingSim_ispass26/output/v6e_8b_baseline/{ds}_bs{N}.csv`.

## Step 3 — Compare native sim output vs real measurements

Built `analysis/compare_v6e_8b.py`. **Pure observation**, no transformation
of sim values beyond arithmetic:

- bs=1: per-request comparison.
  - `real ttft_ms` is reported per-request by the real vLLM measurement
    script (which excludes queue wait — it measures prefill compute only).
  - `sim TTFT` field includes queue wait, so the comparator does
    `(sim.TTFT − sim.queuing_delay)/1e6` to align units.
  - `tpot_ms` and `e2e_ms` analogous.
- bs=2/4: schema mismatch in real CSV — only `batch_e2e_ms` per-batch, no
  per-request TTFT/TPOT. Comparator falls back to total-wall-time vs sum
  of `batch_e2e_ms`. Treat as informational; the two systems use different
  scheduling regimes (real = static batches of N submitted serially; sim =
  continuous batching with all 50 requests in flight).

## Step 4 — DISCARDED experiments (do not reuse)

These were undertaken on 2026-05-04 ~ early 2026-05-05 and **REVERTED**
on 2026-05-05 per user direction. The project's purpose is faithful
reproduction of LLMServingSim v2, not calibration to real measurements.
Listed here so we don't repeat them:

1. **Computed an external scaling factor `sf`** by fitting
   `median(real / sim)` first from bs=1 data (sf≈1.19), then refined
   per-bs via 90 mini-sims (sf={bs=1: 1.19, bs=2: 1.20, bs=4: 1.14}).
2. **Rewrote `layers.csv` and `attention.csv` × sf** in place, replacing
   author-shipped values.
3. **Re-ran sharegpt bs=1** with the rewritten perf data and reported a
   "post-correction" error of ±1.5 % as if the simulator was now accurate.
4. Created scripts `analysis/fit_sf_and_scale.py`,
   `analysis/per_batch_sf.py`, results JSON, mini-sim outputs under
   `LLMServingSim_ispass26/output/per_batch_sf/` and
   `LLMServingSim_ispass26/dataset/per_batch_sf/`.

All artifacts of (1)–(4) deleted on 2026-05-05. Original `layers.csv` and
`attention.csv` restored to author-shipped values.

Side effect: `sharegpt_bs1.csv` (the original sf=1.0 baseline output)
was deleted during step (3) and not preserved. Re-running on 2026-05-05

## Sequence of simulators in one sentence

`python3 main.py` (LLMServingSim, Python) ⟶ spawns `AstraSim_Analytical_Congestion_*`
(astra-sim, C++) as subprocess and pipes per-iteration chakra ET traces (built
from `layers.csv` + `attention.csv` lookups) into it; astra-sim returns cycle
counts that LLMServingSim accumulates into per-request TTFT / TPOT / latency,
and at end writes the result CSV to `output/`.

No other simulator was used. ONNXim was discussed in `memory.md` as a future
cross-check option, but not invoked in this session.
